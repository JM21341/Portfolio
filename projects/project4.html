Table Dictionary

Made by: Taberdo

Database Name: hotel_db

List of Tables

Table Name Description
TBL_BOOKING This table is for reservation transactions made by guests.
TBL_GUEST This table is for storing information about Hotel guests.
TBL_ROOM This table is for the room's information
TBL_PAYMENT This table is for storing payment information.
TBL_STAFF This table is for the hotel employees' information.

TBL_BOOKING

Attributes Data Type Descripption
Booking id int, primary key Unique identifier and primary key for the reservation transaction.
Guest id int, foreign key Foreign key referencing the guest who made the reservation.
Room id int, foreign key Foreign key referencing the specific room reserved
Staff id int, foreign key Foreign key referencing the employee who processed the booking.
Booking status varchar Current state of the reservation.
Check_in_date date The scheduled or actual date the guest arrives.
Check_out_date date The scheduled or actual date the guest vacates the room.

TBL_GUEST

Attributes Data Type Descripption
Guest id int, primary key Unique identifier and primary key for each guest.
Full name varchar Complete combined name of the guest for quick display.
Email unique varchar Electronic mail address for notifications and billing.
Contact number unique varchar Mobile or telephone number of the guest.
Address varchar Physical home or billing address of the guest.

TBL_ROOM

Attributes Data Type Descripption
Room id int, primary key Unique identifier and primary key for each room.
Room type varchar Category of the room.
Room status varchar Current state of the room.
Capacity int Maximum number of guests allowed to occupy the room.
Price decimal Base nightly rate or standard cost for renting the room.
Penalty decimal Late checkout or damage fee associated with the room.

TBL_PAYMENT

Attributes Data Type Descripption
Payment id int, primary key Unique identifier and primary key for each payment.
Booking id int, foreign key Foreign key referencing the corresponding reservation entry.
Payment method varchar Mode of payment used
Payment date date The exact date the payment transaction was processed.
Total amount decimal Total monetary sum paid or owed for the stay.
Days stayed int Total number of nights/days the guest spent in the room.
Payment status varchar Status of the financial transaction.

TBL_STAFF

Attributes Data Type Descripption
Staff id int, primary key Unique identifier and primary key for each staff member.
Full name varchar Complete combined name of the employee.
Position varchar Job title or role assigned to the staff.
Contact number unique varchar Mobile or telephone number of the employee.
Minimum salary daily decimal Minimum baseline daily wage rate for the employee.
Days worked int Number of days rendered by the employee in the current cycle.
Monthly salary decumal Total standard earnings calculated per month. (Minimum salary daily * Days worked)
Entity Relationship Diagram

Made by: Fetalcorin

    Starting off from the top left, an entity named “Guest” is defined. This is the client of the hotel, the customer that the system will take. It contains attributes such as their full name which is split into sub-attributes, their contacts, address, and their very own unique identifier called the “guest_id”. The guests are able to book rooms through the system.

    The Room entity on the upper right represents the room that the guests can stay in. It has a key attribute named “room_id.” It has its own attributes like its type and room number, the capacity, price, and penalty if there are any damages within the guests' stay.

    One Staff member can work with many bookings such as the receptionist and room inspector. A staff must have a full name, position, contact number, and shift schedule. A monthly salary attribute calculates the staff's salary by multiplying their minimum salary daily and the amount of days they worked. There is also a key called “staff_id”, which corresponds to only one staff member for each ID. This entity is essential for running the Hotel Management System because they are the one who processes the data manually.

    Talk about booking, the Booking entity is the main connector for the three entities mentioned above. It has its own key for its own identification for payment later. It contains the room_id, staff_id, and guest_id, which connects all other entities. The booking_status attribute must show if the booking is processed or not, or if it is still in the line. The check in and check out date will be used to calculate the amount of days the guest has stayed. The booking is the very anchor of all these separate entities, which proves its importance in the system.

    Finally, after all the data processing, the guest can finally go on with the payment. The Payment computes the days stayed and the total amount of the guests' stay. It also provides various mediums for the guest to pay, and will notify the guest of its status. This entity is important for managing payment after a guest's stay.

Schema Diagram

Made by: De Ramos

    The presented Schematic Relational Diagram illustrates the database structure of the Hotel Management System designed to manage guests, staff, rooms, bookings, and payment transactions efficiently. The diagram shows how different tables are interconnected through Primary (PK) and Foreign Key (FK) to maintain data integrity and support hotel operations.

    The TBL_BOOKING table records reservation transactions made by guests. It contains the unique booking_id as its primary key and foreign keys guest_id, room_id, and staff_id that link the booking to the corresponding guest, room, and staff member. Additional information such as booking status, check-in date, and check-out date is also stored. This table serves as the central table that connects the major entities within the system.

    The TBL_GUEST table stores information about hotel guests. Each guest is identified using a unique guest_id and includes details such as surname, first name, middle name, full name, email, contact number, and address. This table helps the system maintain records of all guests registered in the hotel.

    The TBL_ROOM table stores information about rooms available in the hotel. Each room is identified by a unique room_id and contains details such as room type, room number, room status, capacity, price, and penalty. This table allows the system to track room availability and accommodation details.

    The TBL_PAYMENT table stores payment information related to bookings. It uses booking_id as both its primary key and foreign key, establishing a one-to-one relationship with the booking table. The table includes payment method, payment date, total amount, days stayed, and payment status. This table ensures that each booking has a corresponding payment record.

    The TBL_STAFF table contains information about hotel employees. Each staff member is identified through a unique staff_id and includes attributes such as surname, first name, middle name, full name, position, contact number, per day salary, monthly salary, days worked, and shift schedule. This table enables the hotel to manage employee records and assign staff members to bookings.

    The relationship between TBL_GUEST and TBL_BOOKING is one-to-many (1:M) because one guest can make multiple bookings, while each booking belongs to only one guest.

    The relationship between TBL_ROOM and TBL_BOOKING is one-to-many (1:M) because one room can be booked multiple times over different periods, while each booking refers to only one room.

    The relationship between TBL_BOOKING and TBL_PAYMENT is one-to-one (1:1) because each booking corresponds to exactly one payment record, and each payment record is associated with only one booking.

    The relationship between TBL_STAFF and TBL_BOOKING is one-to-many (1:M) because one staff member may handle multiple bookings, while each booking is managed by only one staff member.

    Overall, the relational diagram demonstrates the relationships between guests, staff, rooms, bookings, and payment transactions within the Hotel Management System. The design ensures proper organization of hotel data, supports efficient reservation and payment processing, maintains data integrity through primary and foreign keys, and enables effective management of hotel operations.

Data Flow Diagram

Made by: Igig

    The diagram shows how data moves through the Hotel Management System between users, processes, and data storage tables. The system starts with the Registration/Login process, where guests, administrators, room inspectors, and receptionists enter their registration or login information. The system checks these details using the User Table and sends back a confirmation or rejection message depending on whether the information is valid.

    After users successfully log in, they can access the different functions of the system. The Manage Guests process handles guest information by retrieving data from the Guest Table, updating guest records when needed, and saving the changes back to the table. The Manage Rooms process works with room data stored in the Room Table. Room Inspectors can update room status information, and these updates are saved to keep room records accurate and up to date.

    The system also manages reservations and guest stays. The Record Bookings process receives booking updates from the receptionist and stores them in the Reservation/Booking Table. These reservation records are used during the Check-In/Check-Out process to manage guest arrivals and departures. After check-out, payment information is recorded and stored in the Payment Table for future reference.

    Lastly, the system generates reports for management purposes. Information from guest records, room records, bookings, and payment transactions is collected and stored in the Report Table. The Generate Reports process uses this data to create reports that are sent to the administrator. These reports help the hotel monitor its operations and make better decisions. Overall, the system ensures that hotel information is properly managed, updated, and available when needed.

Use Case Diagram

Made by: Fetalcorin

    The diagram above shows the use case diagram of the system. It depicts what each role can do once they access the system. There are 4 entities which have a variety of functions, depending on what are assigned to them. All functions are arranged in colors based on whose function it is. Green for guests, red for receptionist, yellow for housekeeping supervisor, purple for super admin, pink for relation and orange for system.

    Starting off with the Receptionist entity which can login, record booking, view guest details, view receipt, and show the guests' details. The receptionist must login to the system first before accessing their functions. When recording the booking, the receptionist may view the guest details. They can view the receipt and process the payment.

    The Housekeeping Supervisor can view the room's status and record any damages for penalty charges. This entity must login also before they can record inventory and view the room status. If there are any damages, the system will notify the inspector for repair once they view the room status. The housekeeping supervisor will record the damage, and the system will compute the penalty charges, which will be added to the payment.

    The Guest entity is the client which can register/login, book for reservations, and pay for their stay. If the guest is not yet existing, they must register first before logging in. After that, they can book rooms as reservations. After their stay, they can view their receipt and pay for their stay, including the charges.

    Last but not the least, Admin entity — the most important user. The super admin can do all the things the staff can do, including its own functions. They can manage rooms, records, and all users (including guests). They can monitor the system and may generate reports about the status of the system.

Conceptual Diagram

Made by: Ferrer

    The conceptual diagram shows how the Hotel Management System works and how the different users interact with the system from room reservation up to guest check-out.

    The Guest is the customer who uses the hotel services. The process begins when the guest registers an account, books a room, and makes a payment through the system. The guest can also view and receive payment receipts. After the booking is confirmed, the guest is assigned to a room and stays for a specific number of days. Once the reservation period is finished, the guest checks out and leaves the room.

    The Receptionist is responsible for managing guest reservations and assisting guests during the booking process. The receptionist records bookings, checks room availability, views guest details, and processes payments. Any information entered or updated by the receptionist is stored in the database to ensure that guest records remain accurate and up to date.

    The Room Inspector is responsible for checking the condition of hotel rooms before and after guest occupancy. They inspect rooms, update room status, and record inventory information. The room inspector ensures that rooms are clean, complete, and ready for the next guest. All room-related updates are saved in the database for monitoring and management purposes.

    The Admin has the highest level of access in the system. The admin manages rooms, records, and user accounts. They can also generate reports and monitor the overall performance of the hotel management system. Through the system, the admin oversees all operations and ensures that information is properly maintained.

    At the center of the system is the Database, which stores all important information such as guest records, room details, booking information, payment transactions, inventory records, and room status. The receptionist, room inspector, and admin can insert, read, and update data according to their responsibilities. The database serves as the central storage that connects all parts of the system.

    Overall, the process starts when a guest registers and books a room, followed by payment processing and room assignment. The room inspector ensures that the room is available and in good condition, while the receptionist manages guest transactions and reservations. During the guest's stay, all records are maintained in the database. Finally, after the guest checks out, the system keeps the information for future reference and reporting, allowing the hotel to operate efficiently and effectively.
